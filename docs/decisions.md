## 2026-07-02 — Desvio consciente do spec: Google OAuth trocado por e-mail/senha (permanente)

**Desvio explícito e permanente do spec original**, seção 2 ("Autenticação: Google OAuth 2.0"). Decisão do usuário, confirmada verbalmente quando questionado se era temporário ou definitivo ("Permanente — tirar o Google OAuth de vez"). Motivo prático: configurar credenciais reais do Google estava bloqueando qualquer verificação de ponta a ponta do sistema (pendência registrada nas duas entradas anteriores). Este documento é o registro formal desse desvio, para não ficar só implícito no código.

- **"Primeiro usuário a se cadastrar vira Admin" não exigiu nenhuma lógica nova**: `POST /companies` já cria a `Membership(role=ADMIN, status=ACTIVE)` para quem chama, desde a fatia de auth/tenancy original. Só mudou a forma de provar identidade (senha em vez de token do Google) — todo o resto do fluxo (bootstrap de empresa, RBAC, refresh, guards) continua idêntico e não foi tocado.
- **Removidos**: `GoogleStrategy`, `google-profile.interface.ts`, rotas `GET /auth/google`/`GET /auth/google/callback`, dependências `passport-google-oauth20`/`@types/passport-google-oauth20`, campo `User.googleId`, env vars `GOOGLE_OAUTH_*` (local, `.env.example`, `deploy.yml`), e `AuthCallbackPage.tsx`/rota `/auth/callback` no front-end (não existe mais redirect de página inteira — o token nunca mais passa pela URL).
- **`User.passwordHash`** substitui `googleId`. Hash via `node:crypto.scrypt` (salt aleatório por senha, formato armazenado `salt:hash` em hex) — sem dependência nova, mesmo padrão do sha256 já usado no hash do refresh token, mas com uma KDF deliberadamente lenta (`scrypt`), diferente do sha256 do refresh token: senha é segredo de baixa entropia escolhido por humano, refresh token é um segredo de alta entropia gerado pelo sistema — merecem tratamentos diferentes.
- **`POST /auth/register`** (`name`/`email`/`password`, mínimo 8 caracteres) e **`POST /auth/login`** (`email`/`password`) substituem o antigo par `GET /auth/google`/`GET /auth/google/callback`. Resposta agora é **JSON direto**, não mais redirect — login/registro viraram chamadas de API comuns feitas via `fetch` do front-end, não navegação de página inteira vinda do Google. `GET /auth/session`, `POST /auth/select-company/:id`, `POST /auth/refresh`, `POST /auth/logout` não mudaram (operam na camada de token/sessão, independente de como o usuário provou identidade).
- **Migração `basic_auth`** aplicada contra o Cloud SQL real depois de confirmar `users` com 0 linhas (sem risco de perda de dado / violação de `NOT NULL`).
- **Bug real achado só testando no navegador via Playwright** (não nos testes automatizados nem no `curl`): navegar com reload de página inteira (`page.goto`) para uma rota protegida, em vez de navegação SPA, expunha uma corrida em `POST /auth/refresh` — o `React.StrictMode` do modo dev duplica a invocação do `useEffect` de restauração de sessão, disparando duas chamadas de refresh quase simultâneas. Como o refresh token é rotacionado a cada uso (um único uso só), a primeira chamada consome o cookie e emite um novo; a segunda, usando o mesmo cookie já consumido, falha com 401 — e se essa resposta chegar depois da primeira, derruba a sessão que acabou de ser restaurada com sucesso. **Confirmado que é um artefato exclusivo do `pnpm dev`** (StrictMode só duplica efeitos em desenvolvimento, não em builds de produção via `vite build`) — não bloqueou o deploy, mas fica registrado como fragilidade conhecida do mecanismo de rotação de refresh token sob chamadas concorrentes (duas abas, por exemplo, poderiam reproduzir o mesmo problema em produção).
- **Verificação de ponta a ponta via Playwright com registro real** (sem token forjado, pela primeira vez): registrar → criar empresa (confirmado `(ADMIN)` no header) → navegar para Prestadores → logout → login de novo com a mesma senha → sessão restaurada na mesma empresa. Screenshots confirmaram cada etapa.
- **Incidente real de produção durante o deploy desta fatia**: o redeploy da `api` resetou `WEB_ORIGIN` pra `localhost:5173` (fallback de código quando a env var não existe), derrubando CORS pro `web` de verdade. Causa: `--set-env-vars` do `gcloud run deploy` **substitui** todas as env vars da revisão, não faz merge — `WEB_ORIGIN` tinha sido setado manualmente via `--update-env-vars` numa fatia anterior (2026-07-01), fora do `deploy.yml`, então nunca fez parte do `--set-env-vars` do workflow. Corrigido dos dois lados: produção restaurada na hora (`gcloud run services update --update-env-vars WEB_ORIGIN=...`) e `deploy.yml` alterado pra buscar a URL real do `web` e sempre incluir `WEB_ORIGIN` no `--set-env-vars` da `api`, pra não regredir de novo no próximo deploy.

## 2026-07-02 — UI básica: login, seleção/criação de empresa, Prestadores

Primeira UI de verdade no `apps/web`, publicada nas mesmas URLs de sempre. Decisão do usuário: construir a UI agora mesmo sem credenciais reais do Google OAuth ainda (login real continua bloqueado até isso existir).

- **Duas lacunas reais na API, só visíveis testando via navegador** (nunca testadas assim antes, só curl/token forjado):
  1. `GET /auth/google/callback` devolvia JSON — mas o navegador chega lá por navegação de página inteira (redirect do Google), então o usuário veria texto cru na tela. Agora redireciona pra `` `${WEB_ORIGIN}/auth/callback#accessToken=...` `` (token só no fragmento, nunca vai pro servidor/logs).
  2. Cookie de refresh usava `SameSite=Lax`, mas `web` e `api` são serviços Cloud Run em domínios `*.run.app` diferentes — cada `*.run.app` é seu próprio "site" pra regra de SameSite (não é subdomínio de um domínio próprio). `Lax` não é enviado em `fetch`/XHR entre sites diferentes, só em navegação de página inteira — o refresh via `fetch` do front-end simplesmente não funcionaria. Agora usa `SameSite=None; Secure` em produção, mantendo `Lax` sem `Secure` em dev local (HTTPS não existe em `localhost`, `Secure` bloquearia o cookie inteiro).
- **Novo endpoint `GET /auth/session`** (`JwtAuthGuard`, sem exigir empresa) — devolve `companyId`/`role`/`memberships` a partir do token, usado pelo front-end logo após o callback pra decidir pra onde navegar (dashboard, seleção de empresa, ou criação de empresa).
- **Token de acesso só em memória** (`AuthContext`, React) — nunca em `localStorage`/`sessionStorage`, regra não-negociável do spec. Sessão é restaurada a cada carregamento de página via `POST /auth/refresh` (cookie httpOnly, automático via `credentials: 'include'`).
- **Sem biblioteca de UI nova** — HTML semântico simples, reaproveitando o `index.css` já existente.
- **Verificação de ponta a ponta via Playwright headless** (instalado nesta máquina, fora do repositório — `npx playwright install chromium`), navegando com um token forjado no lugar do login Google real: onboarding → dashboard → criar prestador → criar acordo comercial → logout. Screenshots confirmaram renderização correta em cada etapa. Essa é a primeira vez que se usou automação de navegador neste projeto — vale considerar gerar uma skill de projeto (`/run-skill-generator`) se isso for reaproveitado com frequência.
- **Pendência inalterada**: credenciais reais do Google OAuth — sem elas, o botão "Entrar com Google" redireciona e falha no consentimento do próprio Google (client_id inválido), esperado.

## 2026-07-02 — Módulo Contratos (geração + versionamento)

`ContractTemplate` + `Contract`, spec 4.3 — depende de `Provider` (módulo anterior).

- **Avaliador de risco de vínculo fora desta fatia**: o spec coloca isso na fase F3 (Enterprise), separado da geração básica de contrato (F1/MVP). Decisão do usuário: deixar pra quando chegarmos na F3, em vez de fazer uma versão simplificada por palavras-chave agora.
- **Assinatura sem integração real**: `signatureStatus`/`signedAt` só guardam o dado — marcar como assinado é uma ação manual (`PATCH .../signature`). Sem Clicksign, sem upload de PDF (Documentos, spec 4.8, também não existe ainda). Nem interface/stub foi criada ainda para assinatura (diferente do padrão usado com CNPJá) — decisão explícita do usuário de não adiantar essa estrutura nesta fatia.
- **Versionamento reaproveita o padrão de `CommercialAgreement`**: só um `Contract` `ACTIVE` por prestador; gerar um novo marca o anterior `SUPERSEDED` (nunca apaga/sobrescreve) e incrementa `version`. Testado manualmente: segunda geração superou a primeira corretamente.
- **`activityType` em `ContractTemplate` é texto livre, não FK** — a entidade Atividade (spec 4.4) ainda não existe.
- **Renderização de template**: substituição de string simples (`{{provider.legalName}}`, `{{company.legalName}}`, `{{agreement.baseRate}}`, etc.), sem biblioteca de template nova. Placeholders que não resolvem (ou resolvem pra um objeto/array não-stringificável) ficam intactos no texto em vez de virar `[object Object]` — achado real ao lidar com `Decimal` do Prisma no valor do acordo (`agreement.baseRate`), que tem `toString()` próprio mas TypeScript não sabia disso sem tratamento explícito.
- **Bug pego na verificação manual, não nos testes automatizados**: os endpoints aninhados em `/providers/:providerId/contracts/:id` não validavam que o contrato realmente pertencia ao `providerId` da URL — só à empresa (`companyId`). Não era falha de isolamento entre empresas (isso continuava correto), mas a URL "mentia" sobre o que era garantido. Corrigido: `ContractsService.findById`/`updateSignature` agora exigem `providerId` também. Vale o alerta geral: testes unitários com mocks não pegam esse tipo de inconsistência de rota aninhada — só apareceu testando a API de verdade.

## 2026-07-02 — Módulo Prestadores (primeiro módulo de negócio real)

Primeiro módulo do spec seção 4 a ser implementado — `Provider` (Prestador) + `CommercialAgreement` (Acordo Comercial), escopados a `companyId` como todo o resto.

- **Escopo combinado com o usuário**: só cadastro (dados PJ, CPF/RG/endereço) + acordo comercial. Abertura de MEI e conta PJ digital via parceiro (spec 4.2, etapas 2 e 3 do onboarding) ficam de fora — o spec não nomeia esse parceiro, e não está na lista de integrações obrigatórias (CNPJá, PlugNotas, Clicksign, Stark Bank, Resend).
- **Prestador não tem login ainda**: é um registro de dados gerenciado pela empresa (criado só por `ADMIN`), sem vínculo com `User`/`Membership`. Isso fica pra fatia futura de convite/gestão de membros — criar um Prestador não gera automaticamente um jeito dele entrar no sistema.
- **CNPJ único por empresa, não globalmente** (`@@unique([companyId, cnpj])` em `Provider`): o mesmo CNPJ real pode ser prestador de tomadores diferentes, cada um com seu próprio registro — reforça o isolamento multi-tenant em vez de criar um cadastro global de CNPJs compartilhado entre empresas.
- **`address` como campo `Json`**, não colunas separadas — nenhuma integração real (CNPJá, PlugNotas) foi conectada ainda para saber que formato de endereço vai ser exigido; evita comprometer uma estrutura cedo demais.
- **`CommercialAgreement` é histórico, nunca sobrescrito**: criar um novo acordo `ACTIVE` para um prestador que já tem um em vigor encerra automaticamente o anterior (`TERMINATED`, com evento de auditoria) — mesmo padrão já usado em `Membership`. Testado manualmente: criar dois acordos em sequência confirma que o primeiro vira `TERMINATED` com `endDate` preenchida.
- **CNPJá isolada atrás de interface** (`src/providers/cnpj-lookup/`, token de DI `CNPJ_LOOKUP_SERVICE`) — sem credencial de sandbox ainda, a implementação (`StructuralCnpjLookupService`) só confirma o checksum (reaproveita `isValidCnpj`), loga um aviso explícito, e marca o resultado como `source: 'structural-only'`. Trocar pela integração real é só trocar a implementação ligada ao token, sem tocar em `ProvidersService`.
- **Refactor**: `isValidCnpj`/`IsCnpj`, que só existiam em `src/companies/`, foram movidos para `src/common/` — agora `Company` e `Provider` reaproveitam a mesma validação em vez de duplicar.
- Testado manualmente contra o Cloud SQL real (via proxy local): criação de prestador, CNPJ duplicado (409), CNPJ inválido (400), RBAC (`SUPERVISOR` bloqueado de criar, mas pode listar), e a supersessão de acordo comercial.

## 2026-07-01 — Modelo base (Empresa/Usuário/Papel) + autenticação Google OAuth multi-tenant

- **Banco: Cloud SQL real desde já, não Postgres local.** Decisão explícita do usuário (não é a recomendação padrão) — foi criado o projeto GCP `fluxowork` (nº 493843172199, conta `simpleqia.oficial@gmail.com`), habilitado faturamento (conta `FluxoWorkFatt`), e provisionada a instância `fluxowork-dev` (PostgreSQL 17, tier `db-f1-micro`, edição `ENTERPRISE` — a edição padrão do projeto era `ENTERPRISE_PLUS`, que não aceita tiers compartilhados como `db-f1-micro`, por isso `--edition=ENTERPRISE` explícito) em `southamerica-east1`. Banco `fluxowork` criado dentro dela.
  - Conexão local via **Cloud SQL Auth Proxy** (`cloud-sql-proxy.exe`, baixado para o scratchpad da sessão — não faz parte do repo) escutando em `127.0.0.1:5433` (5432 já estava ocupada pelo Postgres nativo desta máquina). Autentica via Application Default Credentials (`gcloud auth application-default login`) — o modo `--gcloud-auth` (reaproveitar login do `gcloud` CLI direto) falhou por causa de uma pasta do Windows com permissão bloqueada (`%APPDATA%\gcloud\legacy_credentials\...`), sem solução encontrada; ADC padrão funcionou.
  - `DATABASE_URL` local aponta para esse proxy: `postgresql://postgres:<senha>@localhost:5433/fluxowork`. A senha do usuário `postgres` foi gerada aleatoriamente na criação da instância e vive só no `.env` local (gitignored).
  - **Pendência:** para rodar a API fora desta máquina (CI, deploy), alguém precisa subir o Cloud SQL Auth Proxy (ou usar IP público autorizado, ou Unix socket no Cloud Run) e ter as próprias credenciais GCP. Isso ainda não está automatizado.
- **Prisma 7 exige adapter de driver explícito — `url` no `datasource` foi removido pela Prisma.** `new PrismaClient()` sem opções (ou com `datasource.url` no schema, que agora é erro de validação) lança `PrismaClientInitializationError`. `PrismaService` agora constrói `new PrismaPg({ connectionString: process.env.DATABASE_URL })` (pacote `@prisma/adapter-pg` + `pg`) e passa via `super({ adapter })`. `prisma.config.ts` continua declarando `datasource.url` normalmente — isso é só para os comandos do CLI (migrate/generate), não para o client em runtime.
- **Output do Prisma Client: local padrão (`node_modules/@prisma/client`), não customizado.** Tentativas de gerar em `apps/api/generated/prisma` (o padrão do `prisma init`) e depois em `apps/api/src/generated/prisma` quebraram o build do NestJS: o `nest build` (tsc) só processa `src/`, então `dist/` não tinha o client, e paths relativos (`../../generated/prisma`) resolviam para lugares diferentes em `src/` vs `dist/`. Voltar para o local padrão do `@prisma/client` resolve via resolução normal de `node_modules`, que funciona igual em qualquer profundidade de diretório — sem esse tipo de bug de path.
- **Modelo de dados**: `Company`, `User`, `Membership` (vínculo usuário↔empresa↔papel, multi-empresa), `RefreshToken`, `AuditLog` — ver `prisma/schema.prisma` em `apps/api`. Detalhes de design (por que `Membership` é entidade separada, por que `AuditLog` nasce nesta fatia, por que sem papel de staff da FluxoWork) ficaram registrados no plano desta fatia, não repetidos aqui.
- **Sessão: JWT de acesso (corpo da resposta, ~15 min) + refresh token opaco (cookie `httpOnly`/`Secure` em produção/`SameSite=Lax`, path `/auth`, hash sha256 no Postgres, ~30 dias, rotacionado a cada uso).** Nenhum dos dois é `localStorage`/`sessionStorage` — o access token vive só em memória no front-end (a implementar). Exigiu `cookie-parser` e `app.enableCors({ origin: WEB_ORIGIN, credentials: true })` (antes era `enableCors()` sem opções, que não suporta cookies cross-origin).
- **Bootstrap de tenant por autocadastro**: `POST /companies` cria a empresa e uma `Membership(role=ADMIN, status=ACTIVE)` para quem chamou, autenticado por um "token de bootstrap" (JWT só com `sub`, emitido no callback do Google quando o usuário ainda não tem nenhum vínculo ativo).

Por decisão do usuário, `.env` local usa um placeholder (`placeholder-client-id.apps.googleusercontent.com`) só para satisfazer essa checagem; `/auth/google` de verdade não funciona até criar um OAuth Client real em console.cloud.google.com/apis/credentials (projeto `fluxowork`, tipo Web application, redirect URI `http://localhost:3000/auth/google/callback`) e substituir os valores. Todo o resto do fluxo (criação de empresa, RBAC, refresh, logout, three-way de guards) foi validado manualmente contra o Cloud SQL real forjando um JWT de bootstrap com o mesmo `JWT_ACCESS_SECRET` do `.env`.

- **apps/api**: gerado via Nest CLI (`nest new`). Bootstrap com `nestjs-pino` (log estruturado, alinhado a Pino + Google Cloud Logging do spec), `ValidationPipe` global (`whitelist`, `transform`), filtro global de exceção (`http-exception.filter.ts`) retornando erro estruturado sem stack trace. Endpoint `GET /health` simples (sem `@nestjs/terminus` ainda — não há dependência externa, como banco, para checar nesta fatia; reavaliar quando Prisma tiver conexão ativa).
- **Prisma**: instalado e configurado (`prisma.config.ts`, `datasource` PostgreSQL) mas **sem nenhum modelo de domínio**. O modelo de entidades (Prestador, Contrato, Ordem de Serviço, etc.) será proposto para validação antes de gerar o schema — ver spec seção 7.
  - Prisma 7 usa `prisma.config.ts` para resolver a `DATABASE_URL` do Migrate/CLI (não mais lida automaticamente do `.env` pelo CLI) — por isso a dependência `dotenv` foi adicionada e importada em `prisma.config.ts`. **Não** usar `output` customizado no `generator client` (testado e revertido — ver fatia 2026-07-01 "modelo base" abaixo): o padrão (`node_modules/@prisma/client`) evita problemas de resolução de caminho relativo entre `src/` e `dist/` depois do build.
  - `pnpm` bloqueia scripts de post-install de pacotes desconhecidos por padrão (supply-chain protection). `unrs-resolver`, `prisma` e `@prisma/engines` foram explicitamente liberados em `pnpm-workspace.yaml` (`allowBuilds` / `onlyBuiltDependencies`) por serem dependências de build necessárias do próprio toolchain (resolver usado por tooling de lint, e o motor nativo do Prisma).
- **apps/web**: gerado via `pnpm create vite` (`react-ts`). Lint via **oxlint** (já vem no template padrão do Vite, rápido, escrito em Rust) em vez de configurar um ESLint flat config do zero — evita duplicar ferramenta de lint sem necessidade. Roteamento com `react-router` (`BrowserRouter` em `main.tsx`). Testes com Vitest + Testing Library (mesma ferramenta de build, sem runner adicional).
- **Sem ESLint compartilhado entre os dois apps**: `api` usa o eslint flat config gerado pelo Nest CLI (ajustado ao ecossistema NestJS/decorators), `web` usa oxlint. Prettier é compartilhado via `.prettierrc` na raiz. Decisão pragmática — forçar uma config única entre um backend NestJS e um frontend Vite tende a gerar mais fricção (regras que não se aplicam a um dos lados) do que valor.
- **CI (`ci.yml`)**: roda lint, typecheck, test, build para os dois apps em push/PR para `main`.
- **Deploy (`deploy.yml`)**: escrito nesta fatia. Ficou não-funcional até 2026-07-01, quando a infra real foi provisionada e a `api` foi hospedada de verdade — ver entrada "API hospedada de verdade no Cloud Run" acima. Disparo continua manual (`workflow_dispatch`), nunca automático em push.
- **Docker**: cada app tem seu próprio Dockerfile multi-stage, com contexto de build na raiz do monorepo (não em `apps/api`/`apps/web`), porque o build depende do lockfile único do workspace. O estágio de runtime do `api` mantém a mesma estrutura relativa de diretórios (`/repo/apps/api`) usada no build, porque o pnpm usa symlinks relativos entre `apps/api/node_modules` e o store em `node_modules/` da raiz — achatar a estrutura quebraria esses links.
- **Sem autenticação/RBAC real nesta fatia** — depende do modelo de dados base (Empresa/Tenant, Usuário, Papel), que é a próxima fatia proposta.

## Pendências conhecidas

- GitHub CLI (`gh`) e autenticação precisaram ser instalados/feitos manualmente nesta máquina (Windows) — não fazem parte do repositório, é ferramenta local.
- `apps/api` e `apps/web` estão hospedados no Cloud Run (ver entradas 2026-07-01 acima).
- Cloud SQL Auth Proxy (`cloud-sql-proxy.exe`) e Google Cloud SDK (`gcloud`) foram instalados nesta máquina fora do repositório — outro desenvolvedor precisa repetir esse setup local (ver seção "Modelo base..." acima) para rodar a API contra o banco real.
- Corrida em `POST /auth/refresh` sob chamadas concorrentes (ver entrada 2026-07-02 "Google OAuth trocado por e-mail/senha") — só confirmado em modo dev (`StrictMode`), mas o mecanismo de rotação de refresh token não tem proteção explícita contra uso concorrente do mesmo cookie. Não é bloqueante, mas vale endurecer se aparecer em produção (ex.: múltiplas abas).
