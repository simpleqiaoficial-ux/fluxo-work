Log curto de decisões relevantes, mais recente no topo. Cada entrada: o que foi decidido, por quê, e o que fica pendente.

## 2026-07-01 — API hospedada de verdade no Cloud Run

`apps/api` está publicamente acessível em **https://fluxowork-api-t4ps6m743a-rj.a.run.app** (a URL muda se o serviço for recriado — conferir com `gcloud run services describe fluxowork-api --region southamerica-east1 --format 'value(status.url)'`). `apps/web` continua sem deploy — próxima fatia.

- **Infra provisionada no projeto GCP `fluxowork`** (região `southamerica-east1`, mesma do Cloud SQL):
  - Artifact Registry: repositório Docker `fluxowork`.
  - Service account de runtime `fluxowork-api-runtime@fluxowork.iam.gserviceaccount.com` — papéis `roles/cloudsql.client` (conectar ao Cloud SQL) e `roles/secretmanager.secretAccessor` (só nos dois secrets abaixo, não no projeto inteiro). É a identidade que o serviço Cloud Run realmente usa.
  - Service account de deploy `github-deployer@fluxowork.iam.gserviceaccount.com` — papéis `roles/run.admin`, `roles/artifactregistry.writer`, e `roles/iam.serviceAccountUser` sobre a SA de runtime (para poder "atuar como" ela ao publicar uma revisão). Usada só pelo GitHub Actions.
  - Workload Identity Federation: pool `github-pool` + provider OIDC `github-provider` (`projects/493843172199/locations/global/workloadIdentityPools/github-pool/providers/github-provider`), restrito a `attribute.repository == "simpleqiaoficial-ux/fluxo-work"`. Sem chave JSON de service account armazenada em lugar nenhum — nem no GitHub, nem localmente.
  - Secret Manager: `fluxowork-api-database-url` (connection string via socket Unix, usuário dedicado `fluxowork_app` — não o `postgres` superusuário usado em dev local) e `fluxowork-api-jwt-secret` (segredo novo gerado para produção, diferente do placeholder local).
  - Usuário de banco `fluxowork_app` criado no Cloud SQL (`fluxowork-dev`), com privilégios `GRANT ALL` só no banco `fluxowork` (tabelas, sequences, e default privileges para tabelas futuras) — não é superusuário.
- **Secrets do GitHub Actions** (repo `simpleqiaoficial-ux/fluxo-work`): `GCP_PROJECT_ID`, `GCP_REGION`, `GCP_WORKLOAD_IDENTITY_PROVIDER`, `GCP_SERVICE_ACCOUNT`, `GCP_ARTIFACT_REPO` — configurados via `gh secret set`.
- **Conexão Cloud Run→Cloud SQL**: conector nativo (`--add-cloudsql-instances`), não o Auth Proxy manual usado em dev local. `DATABASE_URL` de produção usa socket Unix (`?host=/cloudsql/fluxowork:southamerica-east1:fluxowork-dev`), formato diferente do `.env` local (que usa `localhost:5433` via proxy).
- **Bug real encontrado só em produção**: o Dockerfile e o script `start:prod` apontavam para `dist/main.js`, mas o `nest build` compila para `dist/src/main.js` — o `prisma.config.ts` mora fora de `src/`, então o `tsc` calcula um `rootDir` mais alto (a raiz de `apps/api`) e preserva o prefixo `src/` na saída. Isso nunca quebrou localmente porque só usávamos `start:dev` (watch mode do Nest, não `dist/`). Primeira revisão no Cloud Run falhou por causa disso (`Cannot find module '/repo/apps/api/dist/main.js'`); corrigido e redeployado.
- **`gcloud` via bash quebra silenciosamente com valores de flag contendo espaços** (e às vezes aspas duplas), mesmo citados corretamente do lado do bash — é um problema de como o `gcloud.cmd` (wrapper batch do Windows) reconstrói a linha de comando. Solução: usar `--flags-file=arquivo.yaml` (formato `--nome-da-flag: valor`) para qualquer comando `gcloud` com valores que tenham espaço, em vez de passar inline.
- **Pendente**: `GOOGLE_OAUTH_CALLBACK_URL` em produção ainda é placeholder (`https://placeholder/auth/google/callback`) — precisa virar a URL real do Cloud Run quando as credenciais reais do Google forem criadas. `WEB_ORIGIN` da API também não foi setado para nenhuma URL real ainda (não há `web` hospedado). Redis/Memorystore, GCS e as integrações externas (CNPJá, PlugNotas, Clicksign, Stark Bank, Resend) continuam não provisionadas — nenhum código as usa ainda.

## 2026-07-01 — Modelo base (Empresa/Usuário/Papel) + autenticação Google OAuth multi-tenant

- **Banco: Cloud SQL real desde já, não Postgres local.** Decisão explícita do usuário (não é a recomendação padrão) — foi criado o projeto GCP `fluxowork` (nº 493843172199, conta `simpleqia.oficial@gmail.com`), habilitado faturamento (conta `FluxoWorkFatt`), e provisionada a instância `fluxowork-dev` (PostgreSQL 17, tier `db-f1-micro`, edição `ENTERPRISE` — a edição padrão do projeto era `ENTERPRISE_PLUS`, que não aceita tiers compartilhados como `db-f1-micro`, por isso `--edition=ENTERPRISE` explícito) em `southamerica-east1`. Banco `fluxowork` criado dentro dela.
  - Conexão local via **Cloud SQL Auth Proxy** (`cloud-sql-proxy.exe`, baixado para o scratchpad da sessão — não faz parte do repo) escutando em `127.0.0.1:5433` (5432 já estava ocupada pelo Postgres nativo desta máquina). Autentica via Application Default Credentials (`gcloud auth application-default login`) — o modo `--gcloud-auth` (reaproveitar login do `gcloud` CLI direto) falhou por causa de uma pasta do Windows com permissão bloqueada (`%APPDATA%\gcloud\legacy_credentials\...`), sem solução encontrada; ADC padrão funcionou.
  - `DATABASE_URL` local aponta para esse proxy: `postgresql://postgres:<senha>@localhost:5433/fluxowork`. A senha do usuário `postgres` foi gerada aleatoriamente na criação da instância e vive só no `.env` local (gitignored).
  - **Pendência:** para rodar a API fora desta máquina (CI, deploy), alguém precisa subir o Cloud SQL Auth Proxy (ou usar IP público autorizado, ou Unix socket no Cloud Run) e ter as próprias credenciais GCP. Isso ainda não está automatizado.
- **Prisma 7 exige adapter de driver explícito — `url` no `datasource` foi removido pela Prisma.** `new PrismaClient()` sem opções (ou com `datasource.url` no schema, que agora é erro de validação) lança `PrismaClientInitializationError`. `PrismaService` agora constrói `new PrismaPg({ connectionString: process.env.DATABASE_URL })` (pacote `@prisma/adapter-pg` + `pg`) e passa via `super({ adapter })`. `prisma.config.ts` continua declarando `datasource.url` normalmente — isso é só para os comandos do CLI (migrate/generate), não para o client em runtime.
- **Output do Prisma Client: local padrão (`node_modules/@prisma/client`), não customizado.** Tentativas de gerar em `apps/api/generated/prisma` (o padrão do `prisma init`) e depois em `apps/api/src/generated/prisma` quebraram o build do NestJS: o `nest build` (tsc) só processa `src/`, então `dist/` não tinha o client, e paths relativos (`../../generated/prisma`) resolviam para lugares diferentes em `src/` vs `dist/`. Voltar para o local padrão do `@prisma/client` resolve via resolução normal de `node_modules`, que funciona igual em qualquer profundidade de diretório — sem esse tipo de bug de path.
- **Modelo de dados**: `Company`, `User`, `Membership` (vínculo usuário↔empresa↔papel, multi-empresa), `RefreshToken`, `AuditLog` — ver `prisma/schema.prisma` em `apps/api`. Detalhes de design (por que `Membership` é entidade separada, por que `AuditLog` nasce nesta fatia, por que sem papel de staff da FluxoWork) ficaram registrados no plano desta fatia, não repetidos aqui.
- **Sessão: JWT de acesso (corpo da resposta, ~15 min) + refresh token opaco (cookie `httpOnly`/`Secure` em produção/`SameSite=Lax`, path `/auth`, hash sha256 no Postgres, ~30 dias, rotacionado a cada uso).** Nenhum dos dois é `localStorage`/`sessionStorage` — o access token vive só em memória no front-end (a implementar). Exigiu `cookie-parser` e `app.enableCors({ origin: WEB_ORIGIN, credentials: true })` (antes era `enableCors()` sem opções, que não suporta cookies cross-origin).
- **Bootstrap de tenant por autocadastro**: `POST /companies` cria a empresa e uma `Membership(role=ADMIN, status=ACTIVE)` para quem chamou, autenticado por um "token de bootstrap" (JWT só com `sub`, emitido no callback do Google quando o usuário ainda não tem nenhum vínculo ativo).
- **Credenciais reais do Google OAuth: ainda não configuradas.** `passport-google-oauth20` valida `clientID` não-vazio na própria construção do módulo — sem isso a API inteira não sobe (nem para testar rotas que não têm nada a ver com Google). Por decisão do usuário, `.env` local usa um placeholder (`placeholder-client-id.apps.googleusercontent.com`) só para satisfazer essa checagem; `/auth/google` de verdade não funciona até criar um OAuth Client real em console.cloud.google.com/apis/credentials (projeto `fluxowork`, tipo Web application, redirect URI `http://localhost:3000/auth/google/callback`) e substituir os valores. Todo o resto do fluxo (criação de empresa, RBAC, refresh, logout, three-way de guards) foi validado manualmente contra o Cloud SQL real forjando um JWT de bootstrap com o mesmo `JWT_ACCESS_SECRET` do `.env`.
- **Fora desta fatia**: convite de membros por e-mail (Resend), listagem/revogação de membros, troca de papel — ver "Próximo passo" no plano desta fatia.

## 2026-07-01 — F0: fundação do monorepo

- **Gerenciador de pacotes/monorepo: pnpm workspaces**, sem Turborepo/Nx por enquanto. Justificativa: dois apps (`api`, `web`) não justificam o overhead de um orquestrador de build cache ainda; revisitar se o CI começar a ficar lento com o crescimento do número de pacotes.
- **apps/api**: gerado via Nest CLI (`nest new`). Bootstrap com `nestjs-pino` (log estruturado, alinhado a Pino + Google Cloud Logging do spec), `ValidationPipe` global (`whitelist`, `transform`), filtro global de exceção (`http-exception.filter.ts`) retornando erro estruturado sem stack trace. Endpoint `GET /health` simples (sem `@nestjs/terminus` ainda — não há dependência externa, como banco, para checar nesta fatia; reavaliar quando Prisma tiver conexão ativa).
- **Prisma**: instalado e configurado (`prisma.config.ts`, `datasource` PostgreSQL) mas **sem nenhum modelo de domínio**. O modelo de entidades (Prestador, Contrato, Ordem de Serviço, etc.) será proposto para validação antes de gerar o schema — ver spec seção 7.
  - Prisma 7 usa `prisma.config.ts` para resolver a `DATABASE_URL` do Migrate/CLI (não mais lida automaticamente do `.env` pelo CLI) — por isso a dependência `dotenv` foi adicionada e importada em `prisma.config.ts`. **Não** usar `output` customizado no `generator client` (testado e revertido — ver fatia 2026-07-01 "modelo base" abaixo): o padrão (`node_modules/@prisma/client`) evita problemas de resolução de caminho relativo entre `src/` e `dist/` depois do build.
  - `pnpm` bloqueia scripts de post-install de pacotes desconhecidos por padrão (supply-chain protection). `unrs-resolver`, `prisma` e `@prisma/engines` foram explicitamente liberados em `pnpm-workspace.yaml` (`allowBuilds` / `onlyBuiltDependencies`) por serem dependências de build necessárias do próprio toolchain (resolver usado por tooling de lint, e o motor nativo do Prisma).
- **apps/web**: gerado via `pnpm create vite` (`react-ts`). Lint via **oxlint** (já vem no template padrão do Vite, rápido, escrito em Rust) em vez de configurar um ESLint flat config do zero — evita duplicar ferramenta de lint sem necessidade. Roteamento com `react-router` (`BrowserRouter` em `main.tsx`). Testes com Vitest + Testing Library (mesma ferramenta de build, sem runner adicional).
- **Sem ESLint compartilhado entre os dois apps**: `api` usa o eslint flat config gerado pelo Nest CLI (ajustado ao ecossistema NestJS/decorators), `web` usa oxlint. Prettier é compartilhado via `.prettierrc` na raiz. Decisão pragmática — forçar uma config única entre um backend NestJS e um frontend Vite tende a gerar mais fricção (regras que não se aplicam a um dos lados) do que valor.
- **CI (`ci.yml`)**: roda lint, typecheck, test, build para os dois apps em push/PR para `main`.
- **Deploy (`deploy.yml`)**: escrito nesta fatia. Ficou não-funcional até 2026-07-01, quando a infra real foi provisionada e a `api` foi hospedada de verdade — ver entrada "API hospedada de verdade no Cloud Run" acima. Disparo continua manual (`workflow_dispatch`), nunca automático em push.
- **Docker**: cada app tem seu próprio Dockerfile multi-stage, com contexto de build na raiz do monorepo (não em `apps/api`/`apps/web`), porque o build depende do lockfile único do workspace. O estágio de runtime do `api` mantém a mesma estrutura relativa de diretórios (`/repo/apps/api`) usada no build, porque o pnpm usa symlinks relativos entre `apps/api/node_modules` e o store em `node_modules/` da raiz — achatar a estrutura quebraria esses links.
- **Sem autenticação/RBAC real nesta fatia** — depende do modelo de dados base (Empresa/Tenant, Usuário, Papel), que é a próxima fatia proposta.

## Pendências conhecidas

- GitHub CLI (`gh`) e autenticação precisaram ser instalados/feitos manualmente nesta máquina (Windows) — não fazem parte do repositório, é ferramenta local.
- `apps/api` está hospedada no Cloud Run (ver entrada 2026-07-01 "API hospedada de verdade"); `apps/web` ainda não.
- Credenciais reais do Google OAuth (Client ID/Secret) pendentes — tanto local quanto produção usam placeholder; só a criação de empresa/RBAC/sessão foram validados manualmente (não o redirect real do Google).
- Cloud SQL Auth Proxy (`cloud-sql-proxy.exe`) e Google Cloud SDK (`gcloud`) foram instalados nesta máquina fora do repositório — outro desenvolvedor precisa repetir esse setup local (ver seção "Modelo base..." acima) para rodar a API contra o banco real.
